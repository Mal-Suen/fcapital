## 1. 概述 (Overview)

### 1.1 项目背景

传统渗透测试框架存在以下痛点：

1. **工具孤岛**: 各工具独立运行，数据无法自动流转
2. **决策依赖人工**: 每一步操作都需要安全研究员手动判断
3. **知识门槛高**: 需要熟悉大量工具的参数和最佳实践
4. **效率低下**: 重复性操作占用大量时间

### 1.2 项目愿景

构建一个 **AI 驱动的智能渗透测试框架**，实现：

- **自动化决策**: AI 分析每阶段结果，智能推荐下一步操作
- **工具链联动**: 自动选择、安装、执行工具，数据无缝流转
- **人机协作**: 关键决策点由人工确认，AI 处理执行细节
- **知识沉淀**: AI 记忆测试过程，形成可复用的攻击链模板

### 1.2.1 架构决策：混合模式 (Hybrid Mode)

> **关键决策日期**: 2026-04-24
> **决策背景**: 对比"AI分析+工具调度"与"AI直接生成Payload"两种模式

经过架构评估，本项目采用 **混合模式**，而非纯AI生成模式。原因如下：

#### 模式对比分析

| 维度 | AI分析+工具调度 | AI直接生成Payload | 混合模式 (采用) |
|------|----------------|-------------------|-----------------|
| **可靠性** | ✅ 专业工具经过大量验证 | ⚠️ AI生成的payload可能有语法/逻辑错误 | ✅ 标准任务用工具，非标准用AI |
| **可复现性** | ✅ 工具命令可记录、可复现 | ❌ 每次生成可能不同 | ✅ 保留执行记录 |
| **性能** | ✅ 工具针对特定任务优化 | ❌ AI生成代码效率通常更低 | ✅ 各取所长 |
| **法律合规** | ✅ 使用已知工具，责任清晰 | ⚠️ 生成恶意代码的法律边界模糊 | ✅ 审计清晰 |
| **更新维护** | ✅ 工具社区持续更新 | ❌ AI知识有截止日期 | ✅ 工具独立更新 |
| **灵活性** | ⚠️ 受限于已有工具能力 | ✅ 可处理非标准场景 | ✅ AI补充非标准能力 |

#### 混合模式工作流

#### 关键设计原则

1. **AI 不自动执行**: AI 只提供建议，不自动执行下一步操作
2. **用户主导决策**: 所有关键操作由用户选择确认
3. **上下文保留**: 每次操作的结果都保留在上下文中，供后续分析
4. **可回溯**: 用户可以查看历史操作和结果

#### 与自动模式的对比

| 维度 | 自动模式 (旧) | 交互模式 (新) |
|------|--------------|--------------|
| **决策者** | AI 自动决策 | 用户选择，AI 建议 |
| **执行控制** | AI 自动执行 | 用户确认后执行 |
| **风险控制** | 低 | 高 (用户可干预) |
| **适用场景** | 自动化批量扫描 | 专业渗透测试 |
| **可审计性** | 一般 | 高 (每步有记录) |

---

#### 本项目的差异化价值

1. **编排层价值**: AI不是替代工具，而是**决策何时用哪个工具**
2. **上下文管理**: 跨阶段信息关联（信息收集→漏洞发现→利用）
3. **工具生态整合**: nmap/nuclei/sqlmap等各有所长，AI负责调度
4. **可审计性**: 每步操作可追溯，符合渗透测试报告要求
5. **AI辅助生成**: 非标准场景下AI生成定制脚本，填补工具空白

|----------|----------|----------|

## 2. 核心功能需求

### 2.1 AI 驱动的阶段化工作流

#### 2.1.1 功能描述

渗透测试流程被划分为多个阶段，每个阶段结束后，系统将结果格式化发送给 AI，AI 分析后给出下一步建议。

#### 2.1.2 阶段定义

| 阶段ID | 阶段名称 | 输入 | 输出 | AI决策点 |
|--------|----------|------|------|----------|
| PHASE-01 | 信息收集 | 目标域名/IP | 子域名、开放端口、服务指纹 | 选择下一步侦察方向 |
| PHASE-02 | 漏洞发现 | 服务列表 | 漏洞列表 | 评估漏洞优先级 |
| PHASE-03 | 漏洞验证 | 漏洞列表 | 验证结果 | 决定是否深入利用 |
| PHASE-04 | 后渗透 | 获取的访问权限 | 权限提升路径 | 选择持久化策略 |
| PHASE-05 | 报告生成 | 全部测试数据 | 渗透测试报告 | 报告内容优化建议 |

#### 2.1.3 阶段流转规则

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  PHASE-01   │────>│  PHASE-02   │────>│  PHASE-03   │
│  信息收集    │     │  漏洞发现    │     │  漏洞验证    │
└─────────────┘     └─────────────┘     └─────────────┘
       │                   │                   │
       v                   v                   v
   [AI决策]            [AI决策]            [AI决策]
       │                   │                   │
       │              ┌────┴────┐              │
       │              │         │              │
       v              v         v              v
  [跳转/继续]    [跳转/继续]  [跳转/继续]    [跳转/继续]
```

#### 2.1.4 验收标准

- [ ] 支持至少 5 个预定义阶段
- [ ] 每个阶段结束后自动触发 AI 分析
- [ ] AI 响应时间 < 10 秒
- [ ] 支持人工干预跳过/重试阶段

### 2.2 上下文初始化与同步

#### 2.2.1 功能描述

在首次运行时，系统收集本机信息并同步给 AI，作为后续决策的基础上下文。

#### 2.2.2 收集的信息

| 信息类别 | 具体内容 | 用途 |
|----------|----------|------|
| **系统信息** | OS类型、版本、架构 | 选择兼容的工具版本 |
| **网络信息** | 网卡、IP、DNS配置 | 网络扫描参数优化 |
| **已安装工具** | 工具名称、版本、路径 | 避免重复安装 |
| **环境变量** | PATH、代理设置 | 工具执行环境配置 |
| **API密钥状态** | 各服务API配置状态 | 启用高级功能 |

#### 2.2.3 数据格式

```json
{
  "session_id": "uuid-v4",
  "timestamp": "2026-04-22T10:30:00Z",
  "system": {
    "os": "windows",
    "version": "11",
    "arch": "amd64"
  },
  "network": {
    "hostname": "workstation-01",
    "interfaces": [...],
    "dns_servers": [...]
  },
  "tools": [
    {"name": "nmap", "version": "7.80", "path": "C:\\Program Files\\Nmap\\nmap.exe", "status": "ready"},
    {"name": "sqlmap", "version": "1.7", "path": "D:\\Python\\Scripts\\sqlmap.py", "status": "ready"},
    {"name": "hydra", "version": "", "path": "", "status": "missing"}
  ],
  "api_keys": {
    "shodan": "configured",
    "virustotal": "missing"
  }
}
```

#### 2.2.4 验收标准

- [ ] 首次运行自动收集系统信息
- [ ] 信息收集时间 < 5 秒
- [ ] 支持手动刷新上下文
- [ ] 敏感信息（API密钥）脱敏显示

### 2.3 智能工具调度

> **版本**: 2.3.0 重构
> **核心原则**: AI 自由推荐最佳工具，不局限于本地已安装工具

#### 2.3.1 功能描述

AI 根据目标特征和测试需求，推荐**最适合的工具**（而非仅限已安装工具）。当推荐工具未安装时，系统智能处理安装流程。

#### 2.3.2 设计原则

| 原则 | 描述 |
|------|------|
| **最佳工具优先** | AI 推荐最适合当前任务的工具，不受本地限制 |
| **透明化工具状态** | 显示推荐工具的安装状态（已安装/未安装） |
| **智能安装流程** | 工具缺失时，自动尝试安装或引导用户安装 |
| **备选方案机制** | 安装失败时，AI 提供替代工具或手动方案 |

### 2.6 AI 辅助脚本生成 (新增)

> **版本**: 2.1.0
> **添加日期**: 2026-04-24

#### 2.6.1 功能描述

在混合模式下，当遇到非标准场景时，AI 可生成定制化的辅助脚本，填补专业工具的能力空白。

#### 2.6.2 适用场景判断

| 场景类型 | 判断标准 | 执行方式 |
|----------|----------|----------|
| **标准任务** | 有成熟工具支持 | 调用专业工具 (nmap, nuclei, sqlmap等) |
| **非标准场景** | 无现成工具或需定制化 | AI生成辅助脚本 |
| **混合场景** | 需要预处理/后处理 | 工具执行 + AI脚本补充 |

#### 2.6.3 非标准场景示例

| 场景 | 描述 | AI生成内容 |
|------|------|------------|
| 特定CMS漏洞 | 针对某CMS的特定漏洞POC | Python/Bash利用脚本 |
| 定制绕过 | WAF绕过、编码绕过 | 编码转换脚本、payload生成器 |
| 数据处理 | 工具输出格式转换、数据清洗 | 解析脚本、格式转换器 |
| 自动化链 | 多步骤自动化攻击链 | 编排脚本 |
| 特殊协议 | 非标准协议交互 | 协议交互脚本 |

#### 2.6.4 脚本生成流程

```
AI分析阶段结果
        │
        v
┌───────────────────┐
│ 判断是否需要脚本   │
└───────────────────┘
        │
        ├── 不需要 ──> 调用专业工具
        │
        └── 需要 ──> ┌───────────────────┐
                    │ AI生成脚本代码     │
                    │ (Python/Bash/Go)   │
                    └───────────────────┘
                            │
                            v
                    ┌───────────────────┐
                    │ 安全检查          │
                    │ - 代码审计        │
                    │ - 沙箱预执行      │
                    │ - 用户确认        │
                    └───────────────────┘
                            │
                            v
                    ┌───────────────────┐
                    │ 执行脚本          │
                    │ 收集结果          │
                    └───────────────────┘
                            │
                            v
                    ┌───────────────────┐
                    │ 结果融合到上下文   │
                    └───────────────────┘
```

#### 2.6.5 安全控制

| 控制措施 | 描述 |
|----------|------|
| **代码审计** | AI生成代码后，自动进行静态分析，检测危险操作 |
| **沙箱执行** | 在隔离环境中预执行，观察行为 |
| **用户确认** | 关键脚本执行前展示代码，需用户确认 |
| **执行日志** | 完整记录脚本内容和执行结果 |
| **超时控制** | 设置执行超时，防止无限循环 |

#### 2.6.6 支持的脚本语言

| 语言 | 适用场景 | 执行环境要求 |
|------|----------|--------------|
| Python | 复杂逻辑、数据处理 | Python 3.8+ |
| Bash | 简单自动化、命令编排 | Unix-like / Git Bash |
| PowerShell | Windows环境自动化 | PowerShell 5.1+ |
| Go | 高性能、并发场景 | 需编译 |

#### 2.6.7 验收标准

- [ ] AI能识别需要生成脚本的场景
- [ ] 生成的脚本语法正确率 > 90%
- [ ] 脚本执行前需用户确认
- [ ] 脚本执行结果正确融合到上下文
- [ ] 支持脚本保存和复用

### US-001: 初次使用

> 作为一名安全研究员，我希望首次运行时自动检测环境并安装缺失工具，以便快速开始测试。

- 执行 `fcapital init` 后自动收集系统信息
- 列出缺失工具并提供一键安装选项
- 安装完成后显示工具状态摘要

- 输入目标后 AI 自动规划测试步骤
- 每阶段结束 AI 解释发现并建议下一步

- 安装失败时提供备选方案或手动指南

| 术语 | 定义 |
